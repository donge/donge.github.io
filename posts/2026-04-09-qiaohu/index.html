Qiaohu：在骁龙 8 Gen2 上跑一个完全本地的多模态语音助手

9 Apr, 2026
|

最近做了一个小项目 qiaohu，目标很简单：在骁龙 8 Gen2 手机上跑一个完全离线的中文语音助手，不依赖任何云端服务。

整体 pipeline 是：

麦克风 → VAD（Silero）→ LLM（Gemma 4 / LiteRT-LM）→ TTS（sherpa-onnx）→ 扬声器

过程中踩了不少坑，主要集中在 TTS 这块，记录一下。

第一版 TTS：自己写的 Kokoro + ChinesePhonemizer

最开始用的是 Kokoro ONNX 模型配合一套手写的中文音素化器（ChinesePhonemizer）。思路是把汉字转拼音再转 IPA，喂给 Kokoro。

跑起来是能出声，但问题一堆：

PINYIN_TO_IPA 映射表有结构性缺陷。jue、que、xue、yue 这一系列音节全部缺失，导致”觉得”、“学习”、“月亮”这类非常常用的词被静默丢弃——不报错，就是跳过，TTS 输出里这些字消失了。整理下来缺了 25 条音节映射、35+ 个常用汉字。

句间有明显静音。AudioTrack 的写法是先生成完整 PCM 再播放，句子之间必然有间隔，听感割裂。

维护成本高。中文的多音字、数字读法、日期格式……每个边缘情况都要手动处理，永远填不完。

换成 sherpa-onnx

折腾了一阵子之后决定直接换方案。sherpa-onnx 里的 matcha-icefall-zh-baker 模型内置了 espeak-ng 和 FST 文本规范化，数字、日期、标点全都处理好了，根本不需要手写音素表。

输出 22050 Hz Float32 PCM，音质明显好于之前的 Kokoro。

集成方式稍微有点特殊——sherpa-onnx 没有发布 Maven/AAR，需要把 .so 文件手动放到 jniLibs/arm64-v8a/，Kotlin JNI API 直接把源文件复制进项目。大文件全部 gitignore，在 README 里说明从哪里下载。

换完之后 TTS 的问题基本消失了。

遇到的最坑的一个 bug

换完 sherpa-onnx 之后跑起来直接崩，logcat 里是：

java.lang.UnsatisfiedLinkError: OrtGetApiBase

找了很久才定位到：sherpa-onnx v1.12.36 编译时用的是 onnxruntime 1.23.2，但项目 build.gradle.kts 里引用的 onnxruntime-android 是 1.20.0。两个版本的 libonnxruntime.so 符号表不兼容，运行时找不到 OrtGetApiBase。

把 gradle/libs.versions.toml 里的版本号改成 1.23.2 就好了。

教训：用预编译 .so 时一定要确认它编译时依赖的库版本，不要假设”只要大版本一样就行”。

TTS 流式播放

解决了 TTS 本身的问题，句间静音还是存在——因为生成完一整句 PCM 才开始播下一句。

用了一个 Channel<FloatArray> 做 producer/consumer：

  • Producer：对每个句子调用 tts.generate()，生成完就往 channel 里 send
  • Consumer：收到 PCM 立刻写入 AudioTrack 播放

两个协程并发跑，播第一句的同时生成第二句，句间间隔基本消除。

LiteRT-LM 的一些注意事项

LLM 这块用的是 Google 的 LiteRT-LM，跑 Gemma 4 2B。

几个非默认配置缺一不可：

// litertlm-android 用新版 Kotlin metadata 编译，不加这个会报版本不兼容
kotlinOptions {
    freeCompilerArgs += "-Xskip-metadata-version-check"
}

// LiteRT-LM native libs 需要 legacy packaging，否则安装后加载 .so 失败
packaging {
    jniLibs { useLegacyPackaging = true }
}

另外，LiteRT-LM 在 Android 模拟器上会触发 native SIGILL（模拟器不支持 SVE/dotprod 指令），在任何 Java 异常之前进程就挂了。所以代码里加了 isEmulator 检测，模拟器上自动切 stub 模式，方便调试 UI 和 pipeline 逻辑。

目前状态

整体跑通了，效果还不错。延迟主要在 LLM 推理，TTS 本身很快，VAD 响应也比较灵敏。打断功能（barge-in）也实现了：检测到新的语音就 cancel 当前的 ttsJob 并 flush AudioTrack。

代码在 github.com/donge/qiaohu，感兴趣的可以看看。大文件（模型、.so）都没有进 git，README 里有说明从哪里取。
