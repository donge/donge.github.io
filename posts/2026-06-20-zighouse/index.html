ZigHouse: 用 Zig 搓一个 OLAP 引擎

20 Jun, 2026
|

为什么是 Zig

决定用 Zig 写 OLAP 引擎，核心就几点：

  • 零开销抽象是实在的：没有隐式的 allocation、没有 hidden copy、没有 runtime。你写出来的代码就是编译器看到的。
  • 编译期 SIMD：Zig 的 @simd 和显式向量类型让你直接控制 SIMD 宽度，不依赖 runtime dispatch 或 auto-vectorization 的运气。
  • 二进制极小：ReleaseFast 编译，静态链接 LZ4/Zstd，整个引擎不到 3MB。没有 runtime、没有 VM、没有依赖链。
  • 交叉编译开箱即用：zig build -Dtarget=aarch64-linux-gnu 就行，不需要交叉工具链。

还有一条 hidden benefit：Zig 的 comptime 可以让你在编译期做类型特化，比如为不同列类型生成不同的 scan/filter 代码路径，避免虚函数开销。这对列式引擎至关重要——每一行少一次间接调用，乘以千万行就是秒级差异。

站在 ClickHouse / DuckDB 的肩膀上

OLAP 领域最好的两个开源参考就是 ClickHouse 和 DuckDB。ZigHouse 大量借鉴了它们验证过的实践：

  • MergeTree 存储格式兼容：直接读写 ClickHouse 的 .bin / .mrk2 / checksums.txt。这意味着你可以把 ZigHouse 产生的数据目录丢给 ClickHouse 读，反之亦然。
  • ClickHouse 有线协议：实现了 TCP 协议 (端口 9000) 和 HTTP 接口 (端口 8123)。现成的 clickhouse-client、JDBC driver、HTTP 工具链都能直接用。
  • Morsel-driven parallelism：从 DuckDB 学的。一个原子游标分发固定大小的行块给工作线程，不需要显式的 task 调度，cache 局部性好，线程间几乎不竞争。

核心思路是不重复造轮子，但重复造轮子来学习。如果 ClickHouse 已经有做得很好的部分（MergeTree 格式、Wire Protocol），直接兼容比另起一套更有价值。

向量化 / SIMD

列式存储天然适合 SIMD：同一列的数据在内存里连续排列，类型一致，没有分支。

Zig 的做法很直接：

// core/simd_ops.zig — 编译期特化的 SIMD 归约
fn sumVectorized(comptime T: type, data: []const T) T {
    const Vec = @Vector(simdWidth(T), T);
    // 编译期确定向量宽度，生成对应的 SIMD 指令
    var acc = @splat(@as(T, 0));
    for (data[0..data.len - data.len % Vec.len()]) |_, i| {
        acc += @as(Vec, data[i..][0..Vec.len()].*);
    }
    // 处理剩余标量
}

不需要手写 intrinsics，不需要 CPU feature detection，@Vector + @simd 就够了。编译器根据目标 CPU 生成 SSE/AVX/NEON。

关键认知：向量化不止是让 sum 变快。它在 filter、comparison、in-list 里效果更明显——一次比较 8 个值、生成一个 mask、用 mask 做 compress store。这些在 pipeline 执行器里是热路径中的热路径。

整体架构

Parquet / ClickHouse Native blocks
    ↓
Ingest → MergeTree 列文件 (.bin, .mrk2, columns.txt)
    ↓
SQL text → native SQL parser (tokenizer → parser → AST → plan_builder)
    ↓
IR planner → PhysicalNode (11 种 plan shape)
    ↓
Pipeline executor (push-based, morsel-parallel)
    ↓
ResultSet → CSV / ClickHouse Native Block

SQL 解析器是纯 Zig 手写的，不用外部依赖。支持所有 43 个 ClickBench 查询，SQL:2016 符合率 48.9%。

Pipeline 执行器基于 plan shape 做 dispatch：scalar aggregate 走简单的 scan → agg 路径，group by + top N 走 hash agg → topK 路径，复杂查询走多阶段 pipeline。一共 11 种 plan shape，覆盖全部 ClickBench 查询。

性能表现

v1.0.0 在 Apple M2 Pro、10M 行数据上跑完 ClickBench 全部 43 个查询：

  • Warm total: 2.83 秒，0 个 null result
  • 总和耗时是 DuckDB 的 0.75×（已超越 DuckDB）
  • ClickBench 特殊优化版本中排名第一
  • 确定性查询 35/35 结果与 DuckDB 一致

一个人写的引擎，没有几十人的团队、没有 C++ 模板黑魔法、没有几十万行的代码量，能在 ClickBench 上跑到第一，说明技术路线选对了。

构建与分发

zig build -Doptimize=ReleaseFast -Dstrip=true -Dstatic-libs=true

结果是一个 3MB 的静态链接二进制。Homebrew tap、Docker image、GitHub Release 三路分发。交叉编译在 CI 里一步产出 aarch64-macos / x86_64-linux / aarch64-linux。

总结

ZigHouse 谈不上什么创新，就是一个用 Zig 把 OLAP 已知的最佳实践重新实现了一遍的项目。但它验证了几件事：

  • Zig 适合做系统级数据工程，编译期 SIMD + 零开销抽象 + 小二进制，三个点都落在 OLAP 引擎的需求上
  • ClickHouse 的 MergeTree 格式和有线协议设计得很好，兼容比另起炉灶明智
  • 一个人 + 正确的方法论 + 好的工具链，能写出在 ClickBench 上跑进 3 秒的列式引擎

代码在 github.com/donge/zighouse，MIT 协议。
