01 端侧推理的约束 — 为什么难？ background
🖥️
云端 vs 端侧
A100 GPU 有 312 TFLOPS FP16，而手机 SoC 的 NPU 通常只有 10–40 TOPS INT8

云端服务器可以用 8× H100（每张 2000W）堆算力，端侧设备的功耗预算通常是 2–5W，差距高达 3 个数量级。

  • 算力：A100 312 TFLOPS → 手机 NPU ~20 TOPS（INT8）
  • 内存：服务器 80GB HBM → 手机 6–12GB LPDDR5 共享
  • 带宽：HBM 2TB/s → LPDDR5 ~70GB/s
  • 功耗：数据中心无限 → 端侧 2–5W 热设计
📉
核心矛盾
模型参数量指数级增长（GPT-3 175B），而端侧内存只有 GB 级别

一个 7B 参数模型，FP16 精度需要 14GB 内存，端侧完全放不下。但通过量化到 INT4，只需 3.5GB，可以在手机上运行。

这正是端侧推理所有优化技术的根本动机：在约束下尽量保持模型能力。

🔒
端侧的优势
隐私保护、低延迟、离线可用、无网络传输成本
  • 隐私：数据不出设备，符合 GDPR / 个人敏感数据场景
  • 延迟：无网络往返延迟，首 Token 时间可 <100ms
  • 离线：飞行模式、弱网环境正常使用
  • 成本：不产生 API 调用费用，边缘 AIoT 场景尤其重要

02 量化（Quantization）— 最主流的压缩手段 模型压缩

量化将模型权重和激活值从高精度浮点数（FP32）映射到低位宽整数（INT8/INT4），以精度换速度和内存。

各精度格式：内存占用 vs 精度损失（7B 模型为例）
FP32  原始精度28 GB ↑↑↑
FP16 / BF16  训练常用14 GB ↑↑
INT8  推理主流7 GB ↑
INT4 / Q4_K_M  端侧主流3.5 GB ✓
INT2  研究阶段1.75 GB
📐
PTQ — 训练后量化
Post-Training Quantization：直接对已训练模型量化，无需重训练。最常用。

核心公式：Q(x) = round(x / scale) + zero_point

关键在于确定 scale 和 zero_point，常用方法：

  • MinMax：简单，但对 outlier 敏感
  • Percentile：截断极值，更稳定
  • KL 散度校准：最小化量化前后分布差异（TensorRT 用）

需要一小批 calibration data 来统计激活值分布。

🏋️
QAT — 量化感知训练
Quantization-Aware Training：训练时模拟量化噪声，精度更高但需要重训练。

在前向传播中插入 Fake Quantize 算子，使模型学会适应量化误差。反向传播使用 Straight-Through Estimator (STE) 绕过不可微问题。

适用于精度要求严苛的场景（如 INT4 量化），代价是需要 GPU 重训练数小时到数天。

端侧小模型（如 MobileNet）常用 QAT 获得更好的 INT8 精度。

⚡
GPTQ
HOT 基于二阶 Hessian 信息的逐层 PTQ，LLM 4-bit 量化主流方案之一。

核心思路：量化一个权重后，用 Hessian 矩阵信息补偿其他权重，使量化误差最小化。逐列（column-wise）处理，计算量可控。

实际效果：Llama-2-7B 用 GPTQ INT4 量化后，困惑度（perplexity）仅上升 ~0.3，几乎无感知。

pip install auto-gptq model = AutoGPTQForCausalLM.from_pretrained( model_id, quantize_config=GPTQConfig(bits=4) )
🎯
AWQ — 激活感知量化
HOT 保护显著权重（1% 的权重贡献 >10% 精度），比 GPTQ 更快且精度相当。

关键洞察：权重的重要性由对应的激活值大小决定，而非权重本身。只需保护激活值大的通道，即可保住精度。

AWQ 不需要 Hessian 矩阵，速度比 GPTQ 快 3-10×，且不需要 GPU 校准数据。

MLC LLM、llama.cpp 的 GGUF 格式均支持 AWQ 量化导出。

📦
GGUF / Q4_K_M 格式
llama.cpp 的专属量化格式，K-Quant 方法对不同层使用不同精度，端侧 LLM 最常用。

GGUF（GPT-Generated Unified Format）是 GGML 的继任者，在单个文件中打包模型权重、tokenizer、元数据。

K-Quant 命名规则：Q{bits}_K_{size}

  • Q4_K_S：4-bit，小块，速度优先
  • Q4_K_M：4-bit，混合精度，推荐默认
  • Q5_K_M：5-bit，更高精度
  • Q8_0：8-bit，接近 FP16 精度

关键层（如 attention output）保留更高精度，FFN 层用更低精度。

🔀
混合精度推理
不同层使用不同精度：Embedding 层 FP16，FFN 层 INT4，Attention 层 INT8。

核心思路：LLM 各层对量化的敏感程度不同：

  • Embedding 层：词表大，敏感，保持 FP16
  • 第一层/最后层：对误差敏感，用 INT8
  • 中间 FFN 层：数量多，用 INT4 节省最多内存

LLM.int8() 专门处理 outlier 激活值：正常值用 INT8，outlier 列保留 FP16。

03 结构化剪枝 & 知识蒸馏 模型压缩
✂️
结构化剪枝
删除整个 Attention Head、FFN 通道或 Transformer 层，直接减少 FLOPs。

非结构化剪枝（随机置零权重）产生稀疏矩阵，在通用硬件上无法加速，端侧基本没用。

结构化剪枝的目标是删除整块结构：

  • Head 剪枝：某些 Attention Head 贡献极低，可直接删除
  • 通道剪枝：FFN 的部分神经元，按 L1/L2 范数排序删除
  • 层剪枝：删除对输出影响小的整个 Transformer Block

SparseGPT、Wanda 等方法可在不重训练的情况下做结构化剪枝。

🎓
知识蒸馏
大模型（教师）指导小模型（学生）训练，学生学习软标签分布，而非硬标签。

软标签（Soft Label）包含类间相似度信息，比 one-hot 硬标签信息更丰富：

L = α · CE(y, p_student) + (1-α) · KL(p_teacher / T, p_student / T)

T 是温度参数，T 越大，软标签越平滑，类间相似度信息越多。

  • Phi-3-mini：微软用 GPT-4 生成训练数据进行蒸馏
  • Gemma-2B：Google 用 Gemini 蒸馏
  • DistilBERT：BERT 的 40% 体积版本，性能保留 97%
📐
低秩分解（LoRA 原理）
将大权重矩阵分解为两个低秩矩阵之积，LoRA 微调的基础，也可用于推理压缩。

原始权重矩阵 W ∈ ℝ^(d×k)，用两个小矩阵近似：W ≈ A·B，A ∈ ℝ^(d×r)，B ∈ ℝ^(r×k)

当 r ≪ min(d,k) 时，参数量从 d×k 降至 r×(d+k)。

端侧推理时，LoRA adapter 可以动态合并进基础模型权重，不增加推理延迟。

📅
MoE 演进历史
从 GShard 到 DeepSeek-V2，MoE 的规模和效率不断突破。
  • 2020 GShard：首个大规模 MoE，600B 参数，机器翻译
  • 2021 Switch Transformer：简化路由，每 Token 只激活 1 个专家
  • 2022 GLaM：1.2T 参数，推理成本仅是 GPT-3 的 1/3
  • 2023 Mixtral 8×7B：激活参数仅 13B，性能超 Llama-2-70B
  • 2024 DeepSeek-V2：Fine-grained MoE + 共享专家，236B 总参/21B 激活
📱
端侧 MoE 的挑战
MoE 减少 FLOPs 但不减少内存占用，端侧需要 Expert Offloading 解决。

关键矛盾：Mixtral 8×7B 虽然激活参数只有 13B，但需要加载全部 47B 参数到内存。

Expert Offloading：将不活跃的 Expert 存到 CPU 内存或闪存，按需换入 GPU/NPU。

  • PowerInfer：利用 LLM 激活的局部性，将热门 Expert 常驻 GPU
  • MLC LLM：支持 MoE 的量化与分层 offload

Expert 路由的不规则性对硬件不友好，是端侧 MoE 工程难点。

05 位置编码演进 架构

Transformer 本身无序列位置感知，位置编码的设计直接影响模型的长上下文推理能力——这在端侧尤其重要，因为内存是瓶颈。

2017 · Vaswani et al.
正弦绝对位置编码（APE）
PE(pos, 2i) = sin(pos/10000^(2i/d))。固定公式，不可学习，无法外推到训练长度之外。原始 Transformer 标配。
2018 · BERT
可学习绝对位置编码
每个位置一个可训练向量，表达能力更强，但同样无法外推。BERT 最大位置 512。
2019–2021 · T5 / Shaw et al.
相对位置编码（RPE）
编码的是 token 间的相对距离而非绝对位置，具备一定外推能力。T5 用 bucket 量化的 bias，计算在 Attention logits 上加偏置。
2021 · 苏剑林（Su et al.）
RoPE — 旋转位置编码 ★
将位置编码融入 Q/K 的旋转变换：q_m · k_n 的内积只依赖相对位置 (m-n)。 外推能力好，计算高效，成为 Llama / Qwen / Mistral 等几乎所有现代 LLM 的标配。
2022 · Press et al.
ALiBi — 线性偏置注意力
不添加位置向量，在 Attention 分数上减去与距离成正比的线性惩罚。可在短序列训练后推理更长序列，MPT 模型使用。
2023–2024
RoPE 长度外推：YaRN / LongRoPE / RoPE Scaling
端侧直接关联：更长上下文 = 更大 KV Cache = 更多内存。 YaRN 通过频率插值让 4K 训练的模型推理 128K；LongRoPE 动态调整旋转角度。 端侧常用线性/NTK 插值 scaling 因子简单扩展上下文窗口。
06 KV Cache 演进 内存优化

自回归解码中，每生成一个 Token 需要访问所有历史 Token 的 K/V 矩阵。KV Cache 是端侧 LLM 最大的内存消耗来源。

KV Cache 内存计算（FP16）
Memory = 2 × num_layers × seq_len × num_heads × head_dim × 2 bytes
例：Llama-3-8B（32层，32头，128维），seq=4096 → KV Cache = 2×32×4096×32×128×2 ≈ 2GB
方案 时间 K/V Head 数 内存节省 精度影响 端侧支持
MHA（原始） 2017 H 个 K，H 个 V 基准 无损失 内存压力大
MQA 2019 1 个 K，1 个 V 节省 H× 轻微下降 极友好
GQA 2023 G 个 K，G 个 V 节省 H/G× 接近 MHA Llama-3 标配
Sliding Window 2023 H 个，固定窗口 固定上限 长依赖受损 Mistral 使用
PagedAttention 2023 分页管理 碎片减少 无损失 参考借鉴
量化 KV（INT8） 2023+ 不变 节省 2× 极小下降 llama.cpp 支持
量化 KV（INT4） 2024 不变 节省 4× 可接受 MLC LLM 支持
🔍
MQA vs GQA 原理
MQA 所有 Query Head 共享同一组 KV，GQA 分组共享，两者折中。

MHA：每个 Query Head 有独立的 K, V → KV Cache 大小 = H × seq × d

MQA：所有 H 个 Query Head 共享 1 组 KV → KV Cache 大小 = 1 × seq × d（减少 H 倍）

GQA：H 个 Query Head 分为 G 组，每组共享 1 组 KV → KV Cache = G × seq × d

Llama-3-8B 用 GQA，32个 Q Head，8个 KV Head（G=8），KV Cache 减少 4×。

📄
PagedAttention
借鉴操作系统虚拟内存分页思想，将 KV Cache 切成固定大小的 Page 管理。

传统 KV Cache 预先分配连续内存，导致大量内部碎片（平均浪费 60-80%）。

PagedAttention 将 KV Cache 分为固定大小的 Block（如 16 个 Token），用逻辑-物理地址映射管理，内存利用率提升至 >95%。

vLLM 的核心技术，端侧可借鉴其 Block 管理思想，在内存受限场景动态管理长序列。

算子融合（Kernel Fusion）
将多个连续算子合并为一个 Kernel，减少内存读写往返（memory roundtrip）。

典型案例：Attention 中的 Linear + Bias + GELU 三步可合并为一次 Kernel 调用。

每次 Kernel 调用都涉及：写显存 → 读显存 → 计算 → 写显存。融合后只需最终写一次。

  • LayerNorm + 线性层融合：Transformer 最常见的融合
  • QKV 投影融合：3 个 Linear 合并为 1 个大矩阵乘
  • Attention + Softmax + Dropout：Flash Attention 的融合策略
📉
常量折叠 & 死代码消除
推理时某些节点输入永远是常量，提前计算并内联，减少运行时开销。

常量折叠：如 LayerNorm 的 weight/bias 是常量，可以预计算并与前一层融合。

死代码消除：训练时用到的 Dropout（推理时 p=0）、某些辅助输出节点，推理时删除。

ONNX Runtime 的图优化 Pass 会自动做这些变换，MNN/NCNN 也内置类似优化。

🧮
GEMM 优化：Tiling & Pack
矩阵乘法（GEMM）是 LLM 推理 >90% 的计算量，Tiling 提升缓存命中率。

Tiling（分块）：将大矩阵切成能放进 L1/L2 Cache 的小块，每块在缓存中完成计算，避免大量 Cache Miss。

Pack：将矩阵 B 重排内存布局，使连续访问对应连续内存，提升向量化效率。

// 典型 Tiling 伪代码 for (int i = 0; i < M; i += TILE_M) for (int k = 0; k < K; k += TILE_K) for (int j = 0; j < N; j += TILE_N) // TILE_M × TILE_K × TILE_N 的小块乘法 // 全部在 L1 Cache 中完成
📈
Roofline Model
判断计算是「算力受限」还是「带宽受限」的分析工具，指导优化方向。

算术强度（Arithmetic Intensity）= FLOPs / Bytes（计算量/访存量）

如果算术强度 < 设备 Ridge Point（峰值算力/峰值带宽），则是带宽受限（Memory-Bound）。

  • Decode 阶段：batch=1，矩阵乘退化为矩阵-向量乘，严重 Memory-Bound
  • Prefill 阶段：batch 较大，Compute-Bound，GPU 利用率高

端侧 Decode 基本是 Memory-Bound，所以量化（减少带宽）比增加算力更有效！

08 内存规划 — 推理引擎的静默优化 引擎工程
📊
Tensor 生命周期分析
推理是有向无环图（DAG），每个 Tensor 有确定的「产生」和「最后使用」时刻。

如果两个 Tensor 的生命周期不重叠，就可以复用同一块内存。

这等价于图着色问题：最小化颜色数（内存块数）。

// 示意：Layer0 输出在 Layer2 输入后可以释放 // Layer3 输入可以复用 Layer0 输出的内存 Tensor A: [0, 3] // 算子 0 产生，算子 3 最后使用 Tensor B: [1, 2] // 可以和 A 共用内存（生命周期不重叠） Tensor C: [4, 6] // 可以复用 A 的内存
♻️
In-place & Alias 优化
某些算子（如 ReLU、Reshape）可以直接在输入内存上原地操作，无需分配新内存。
  • In-place：激活函数、Add、LayerNorm 等可原地操作
  • Alias：Reshape / Transpose / Slice 可以共享底层数据，只修改 strides
  • Concat 优化：提前分配输出内存，各输入直接写入对应偏移位置

NCNN 的 Blob 设计、TFLite 的 arena 分配器都大量使用这些技术。

🗄️
静态 vs 动态内存分配
静态分配：编译时确定内存布局，零运行时开销；动态分配：支持变长输入。

静态分配：固定输入尺寸（如 MobileNet 224×224），编译时做 lifetime 分析，生成固定内存池。TFLite 默认策略，适合嵌入式。

动态分配：LLM 序列长度不固定，需要运行时分配。代价是内存碎片和分配开销。

混合策略：权重用静态（加载时分配），KV Cache 和激活值用动态（带内存池）。

09 Flash Attention & 投机解码 计算优化
⚡
Flash Attention
IO-aware 的注意力实现：将 QKV 分块，在 SRAM 中完成计算，避免 O(N²) HBM 读写。

标准 Attention 的瓶颈不是计算量，而是 HBM 读写：N×N 的 score 矩阵需要写入再读出 HBM，随序列长度平方增长。

Flash Attention 核心：Tiling + Online Softmax。将 Q/K/V 分成小块，每块在 SRAM（片上高速内存）中完成 Attention 计算，结果累积，不写中间矩阵到 HBM。

  • Flash Attention 1：2022，速度提升 2-4×，内存 O(N)
  • Flash Attention 2：2023，减少 non-matmul FLOPs，速度再提升 2×
  • Flash Attention 3：2024，专为 H100 Tensor Core 优化

端侧：Apple Metal / ARM NEON 均有对应实现（llama.cpp 内置），NPU 的 SRAM 特性同样适合 Flash Attention 策略。

🔮
投机解码（Speculative Decoding）
HOT 用小模型快速草拟多个 Token，大模型并行验证，拒绝则回退。不改变输出分布。

动机：自回归解码每步只生成 1 个 Token，大模型严重低效（Decode 阶段是 Memory-Bound）。

流程：

  • 1. 小模型（Draft Model）自回归生成 γ 个候选 Token
  • 2. 大模型（Target Model）对 γ+1 个位置并行做一次前向
  • 3. 用拒绝采样（Rejection Sampling）验证，不改变大模型分布
  • 4. 平均接受 β 个 Token（β ≈ 2-4×），相当于 2-4× 加速

端侧变体：Self-Speculative Decoding（同一模型的不同层做 Draft），无需额外小模型。

10 线程模型与并行调度 引擎工程
🔀
Intra-op 并行
单个算子内部的并行：将矩阵乘按行/列切分，多线程分别计算。

对于 GEMM（M×K × K×N），可以沿 M 维切分，每个线程计算一部分行：

// 4 线程并行，每线程计算 M/4 行 thread[i]: C[i*M/4 : (i+1)*M/4, :] = A[...] × B

llama.cpp 的 ggml_compute_forward 用线程池 + barrier 同步实现 intra-op 并行，是其性能核心。

⏭️
Inter-op 并行
多个无依赖算子同时执行（流水线并行），减少总执行时间。

计算图中有些节点无数据依赖（如 MoE 中不同 Expert），可以并行执行。

实现上用有向图拓扑排序 + 就绪队列，就绪节点分发到空闲线程。

端侧效果有限，因为 Decode 阶段计算图大多是线性的。但 Prefill 阶段有更多并行机会。

📱
大小核调度（big.LITTLE）
Android / ARM 设备有高性能大核和低功耗小核，推理引擎需要合理分配任务。

Android：通过 pthread_setaffinity_np 或 QoS API 绑定大核。

Apple：通过 QOS_CLASS_USER_INTERACTIVE 让系统自动调度到 P-Core（性能核）。

  • 性能核（P-Core）：执行计算密集型 GEMM
  • 效能核（E-Core）：处理 IO、KV Cache 加载、预处理

llama.cpp 在 Apple Silicon 上会优先占满 P-Core，同时利用 Metal GPU。

ONNX 微软/Facebook Protobuf INT8（QDQ） 通用 IR，跨框架转换

🏗️
后端抽象设计
推理引擎需要在同一套 API 下支持 CPU / GPU / NPU 多种后端。

核心模式：Operator Registry + Backend Interface

// 典型后端抽象（伪代码） class Backend { virtual void matmul(Tensor& C, Tensor& A, Tensor& B) = 0; virtual void softmax(Tensor& out, Tensor& in) = 0; }; class CPUBackend : public Backend { ... }; class MetalBackend : public Backend { ... }; class HexagonBackend : public Backend { ... };

算子注册表：每个算子对多个后端分别注册实现，运行时根据设备能力选择。

🔄
Fallback 机制
NPU 不支持的算子自动 fallback 到 GPU，GPU 不支持的 fallback 到 CPU。

实际中 NPU 只能高效运行固定算子集合（Conv/MatMul/Sigmoid），不规则算子（如 RoPE 的复数运算）需要 fallback。

频繁 CPU↔GPU 数据搬运会抵消 NPU 加速，因此 Fallback 策略需要权衡：

  • 子图分割：将 NPU 支持的部分切成子图，整块送 NPU
  • 算子融合后再判断：融合后的算子可能整体由 NPU 支持

13 硬件加速 — SIMD / NPU / 异构调度 硬件
🔢
SIMD 指令集演进
Single Instruction Multiple Data：一条指令同时处理多个数据，是 CPU 推理加速的基础。
  • x86 SSE（1999）：128-bit，4×FP32 同时计算
  • x86 AVX2（2013）：256-bit，8×FP32 或 8×INT32
  • x86 AVX-512（2017）：512-bit，16×FP32，服务器主流
  • ARM NEON（2004）：128-bit，移动端 SIMD 标配
  • ARM SVE（2016）：可变长度向量，MacBook M 系列 512-bit
  • ARM SVE2（2019）：增强整数支持，INT4 量化加速

llama.cpp 针对 NEON/AVX2/AVX-512 分别有专门的量化矩阵乘实现，性能差异 3-5×。

🧠
NPU / DSP 架构
专为 AI 矩阵乘设计的加速单元，通常比 CPU 高效 10-50×（INT8）。
  • Apple Neural Engine（ANE）：A11 起集成，M 系列最强 ~38 TOPS，支持 Core ML 直接调用
  • 高通 Hexagon NPU：Snapdragon 8 Gen 3 达 ~73 TOPS，QNN SDK
  • 联发科 APU：Dimensity 9300 ~35 TOPS，NeuroPilot SDK
  • 三星 NPU：Exynos 2400 ~14.7 TOPS

NPU 通常只支持固定精度（INT8/INT16）和固定算子集，灵活性低于 GPU，需要专门优化。

🎮
移动 GPU（Shader 计算）
Adreno / Mali GPU 通过 Compute Shader（OpenCL / Vulkan / Metal）执行通用计算。

移动 GPU 与 CPU 共享内存，避免 PCIe 数据搬运，延迟低。

  • Metal（Apple）：llama.cpp Metal 后端，M2 Pro 上 ~30 token/s（7B Q4）
  • Vulkan：跨 Android 设备，MLC LLM / MNN 均支持
  • OpenCL：NCNN 最早的 GPU 后端，兼容性最广

端侧 GPU VRAM 和主存共享，无显存容量限制，但带宽低于离散 GPU。

🔀
异构调度策略
CPU + GPU + NPU 协同：用计算特性决定谁来执行哪部分。

典型策略：

  • Prefill（长输入处理）：计算密集，送 GPU / NPU
  • Decode（逐 Token 生成）：Memory-Bound，CPU + GPU 均可
  • 不规则算子（RoPE / TopK）：CPU 处理，避免 GPU 调度开销
  • 量化权重解压：CPU 解压后送 GPU，或 GPU Shader 内解压

Core ML 的 MLComputeUnits.all 让系统自动在 ANE/GPU/CPU 间调度，是目前最完善的异构实现。

🍎
Apple Instruments
Xcode 内置，Time Profiler + Metal System Trace，分析 CPU/GPU/ANE 使用。
  • Time Profiler：CPU 热点函数，采样粒度 1ms
  • Metal System Trace：GPU Shader 执行时间线，找 Shader 瓶颈
  • Core ML Instrument：ANE / GPU / CPU 分别的耗时，自动标注调度决策
  • Memory Graph：内存分配可视化，找内存泄漏和峰值
📱
Android Profiling
Android Studio Profiler + Snapdragon Profiler，覆盖 CPU/GPU/DSP。
  • Android Studio CPU Profiler：采样/插桩两种模式，方法级耗时
  • Snapdragon Profiler：Adreno GPU 专用，Shader 分析，带宽使用率
  • simpleperf：命令行 perf，可分析 JNI / NDK 层
  • Perfetto：系统级 trace，多进程时间线，分析调度延迟
📏
关键指标
端侧 LLM 推理的核心 KPI：TTFT、Decode throughput、内存峰值。
  • TTFT（Time To First Token）：Prefill 延迟，决定用户感知响应速度
  • Decode throughput：token/s，决定生成流畅度（>15 token/s 较流畅）
  • 内存峰值：模型权重 + KV Cache + 激活值，决定是否 OOM
  • 功耗：长推理任务的电池消耗，端侧独特指标
  • 模型精度：量化/压缩后的 perplexity / benchmark 分数回归
